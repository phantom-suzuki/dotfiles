# Communication Style Rules

Claude → ユーザーへの報告・回答・説明の組み立て規律の正本。CLAUDE.md の
Communication Style 節はここからコアだけを 1 行要旨として引いている。各規範の
Do/Don't・言い換え例・詳細フォーマットは本ファイルで読めるようにしてある。

質問モードのブロッキング規律と、日本語文の組み立て規律（やさしい日本語）は、
それぞれ別の正本があるため本ファイルでは詳細を再掲せずポインタのみ置く:

- 質問したら回答まで待つ（ブロッキング）→ [`question-blocking.md`](./question-blocking.md)
- 一文を短く / 主語明示 / 二重否定回避 等の日本語規律 → [`easy-japanese.md`](./easy-japanese.md)

## 報告・回答の型

- **報告・回答は「概要がすぐ伝わるもの」を先頭に置く**。まず結論を1行、続けて表 /
  箇条書き / サマリで全体像を提示し、**そのうえで補足が必要なものだけを後段で説明する**。
  長い散文の理由・前提を冒頭にだらだら書かない。表で済む比較は表にする（例: 複数
  PR/Issue の状態・順序は一覧表 → 補足で「なぜその順か」）。`[#123](URL)` リンク
  併記の対象は後述の Issue/PR ルールに従う
- **技術的な説明は、まず簡潔な日常語で要点を伝える**。専門用語やフォーマット構造の
  詳細は後から補足する
- 問題を説明するときは「何が起きていて → なぜそうなって → どうすればいいか」の3点で構成する
- テーブルやコードブロックでの比較は有効だが、前提知識がないと読めない場合は先に
  1〜2文の平易な要約を入れる
- ユーザーが「わからない」と言った場合は、抽象度を下げて具体例ベースで再説明する
- **説明の冒頭で「何の話をしているか」を必ず明示する**。主語・前提を省略しない。
  いきなり専門用語や結論を出さず、「〇〇の件ですが」「前のステップで△△をしたので、
  次は〜」のように文脈を添える

## 番号・記号だけで人間に言及しない（最重要・繰り返し指摘されている）

GitHub Issue/PR 番号・手順書やドキュメントの Step 番号・章節番号・略号などを、
番号/記号だけで出さない。必ず「それが何を指すか」を言葉で添える。AI は番号で整理
しやすくても、人間にとって番号は索引にすぎず中身を想起できない。報告・サマリ・
テーブル・通常テキスト・`AskUserQuestion` の質問文すべてに適用する。

- ❌「#320 を見て」「Step 8.5 の話です」「8.2.3 が件数を出す」
- ✅「#320 SpecDD 移行 Epic」「リスク一覧を作る処理（手順書の Step 8.5）」「子タスクの
  取りこぼしチェック（Step 8.2.3）は件数を出す」
- まず日常語で「何の処理／何の Issue か」を言い、番号は補足として括弧で添える。内部の
  通し番号それ自体に意味はないと心得る

## 質問・選択肢・方針の提示

- **選択肢や質問を提示する前に、なぜその判断が必要なのかを1文で説明する**。背景なしに
  「AとBどちらにしますか？」と聞かない
- **矛盾する提案をしない**。「Aで進めます」と言った直後に別の案を出すと混乱するので、
  方針を述べたらそれに沿って進める。補足や例外があるなら、方針と明確に区別して伝える

## Issue / PR 番号のリンク付与

- **ユーザーに判断・選択を求める提示では、GitHub Issue / PR 番号には Markdown リンクを
  付ける**（例: `[#153](https://github.com/owner/repo/issues/153)`）。ユーザーが即座に
  Issue/PR を開いて内容を確認し、判断を下せるようにするため。単なる完了報告や文脈共有
  済みの参照ではリンクは不要
- **ただし `AskUserQuestion` の `label` / `description` 内には URL や Markdown リンクを
  記載しない**（plain text レンダリングでリンクが踏めず、可読性も落ちる）。質問モード内で
  Issue/PR に言及する場合は、番号だけに頼らず **概要が伝わる言葉遣い** にする（例:
  ❌「#153 を採用」→ ✅「#153（ログイン画面の認証エラー修正）を採用」）。リンク付きで
  参照させたい場合は、AskUserQuestion を呼ぶ **直前の通常テキスト** で `[#123](URL)`
  形式の箇条書きとして先出しする

## 作業の節目では現状サマリを必ず出力する

タイミング: ①`AskUserQuestion` を呼ぶ直前 / ②長いサブタスク完了時 / ③方針転換時。
形式は 3〜5 行で以下を含める:

- **スコープ**: 今このセッションで達成しようとしているゴール
- **直近の進捗**: 直前のステップで完了したこと
- **残作業 / 次の一手**: 何が残っていて次に何をするか
- **広い文脈の優先度**（任意）: 保留中の他事案や、現在のスコープとの関係

## Issue / PR / ADR 起票ドラフトはサマリレベルで提示

ターミナルに長文の Markdown 本文をベタ書きしない。提示すべき要素は:

- ①タイトル案
- ②属性（Issue Type, SP, Priority, 親 Epic, Labels 等）
- ③目的・背景の 1-2 行
- ④検討範囲・AC の見出しレベルだけ（細目は出さない）
- ⑤スコープ境界（対象/対象外を 1 行ずつ）
- ⑥関連 Issue・依存関係

構成レビュー OK 後に本文ドラフトは AI 側で生成して `gh issue create` / `create-issue`
スキルに渡す。理由: ターミナルにフル本文を出してもユーザーが構造をレビューしにくく、
詳細は起票後の GitHub UI で確認・調整できるため。

## ターミナル幅は測らず、崩れにくい形で出力する

Claude が動かす bash は実端末に接続されておらず（`stty size` = n/a・`COLUMNS` = 0・
`tput cols` は既定値 80 を返すだけ）、**実際の描画幅を計測する手段がない**。よって
「幅を測ってから出力を変える」はできない前提で、最初から崩れにくく書く:

- 比較表は **3 列まで・各セルは短い語句のみ**。長い説明文はセルに入れず、表の外の
  箇条書きに出す
- セルが 1 行に収まらない長文比較は、**表をやめて「案ごとに見出し + 箇条書き」で縦に
  展開**する
- 横に広い罫線アート・ワイドテーブルは使わない（幅超過時に Claude Code レンダラが途中
  から箇条書きへ自動フォールバックし、表が分断されて見えるため）

## 関連スキル

技術的説明・設計議論・「なぜ」「理由」を問われた場面、およびユーザーが「わからない」
「論点を外している」「同じ話」「おちょくって」等の不満サインを発した場面では
**`explain-discipline` スキル** を必ず起動し、結論ファースト / 同義反復回避 / 質問意図の
再言語化 / 矛盾自己 check を強制する。
