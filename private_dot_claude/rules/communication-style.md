# Communication Style Rules

**適用範囲**: Claude → ユーザーへの**すべての日本語出力**（チャット応答 / Issue / PR / コメント / 説明 / 報告サマリ / プラン文）に常時適用する。技術用語のカタカナ語・コード識別子・コミットメッセージの英語部分は対象外。

3 つの文体規範を 1 本に統合したもの（報告・説明の型 = 旧 `communication-style` / やさしい日本語 R1〜R8 = 旧 `easy-japanese` / 質問ブロッキング = 旧 `question-blocking`）。各規範は「規範 1 行 + 例 1 つ」に圧縮。

## 報告・回答の型

- **概要ファースト**: まず結論 1 行、続けて表 / 箇条書きで全体像。長い前提を冒頭に書かない。例: 複数 PR の状態は一覧表 → 補足で「なぜその順か」
- **技術説明は日常語から**: まず平易な要点、専門用語・構造は後から補足する
- **問題説明は 3 点構成**: 何が起きて → なぜ → どうするか
- **冒頭で「何の話か」明示**: 主語・前提を省略しない。例:「〇〇の件ですが」「前ステップで△△したので次は〜」
- **「わからない」と言われたら**抽象度を下げ、具体例ベースで再説明する

## 番号・記号だけで人間に言及しない（最重要）

Issue/PR 番号・Step 番号・章節番号・略号を、番号/記号だけで出さない。必ず「何を指すか」を言葉で添える。

- ❌「#320 を見て」「Step 8.5 の話です」
- ✅「#320 SpecDD 移行 Epic」「リスク一覧を作る処理（手順書の Step 8.5）」

## 質問・選択肢・方針の提示

- **選択肢の前に、なぜその判断が必要かを 1 文**添える。背景なしに「A と B どちら?」と聞かない
- **矛盾する提案をしない**。「A で進める」と言ったらそれに沿う。例外があるなら方針と明確に区別する
- **設計選択肢は「比較表 → 選択 → 詳細」の順**。全案の詳細を一気に並べて判断を麻痺させない

## 質問モードのブロッキング（最重要）

`AskUserQuestion` で質問を出したら**ユーザーの回答まで待つ**。時間がかかっても待ち続ける。

- タイムアウト（`No response after Ns ...`）を、離席・省略・自己判断の許可と**解釈しない**
- best judgment で先に進めない。回答を待つ。待機中は読み取り専用の下調べのみ可（答えに依存する判断・変更・外向き操作には着手しない）
- 例外: ユーザーが事前に「任せる」「デフォルトで」等と**明示的に**委任した場合のみ、デフォルトを選んで進めてよい

## Issue / PR 番号のリンク付与

- ユーザーに判断・選択を求める提示では Issue/PR 番号に Markdown リンクを付ける（例: `[#153](URL)`）。単なる完了報告・文脈共有済みの参照では不要
- **`AskUserQuestion` の label / description 内には URL / リンクを書かない**（plain text で踏めない）。番号でなく概要が伝わる言葉遣いにする。リンク付き参照は質問直前の通常テキストで先出しする

## 作業の節目で現状サマリを出す

タイミング: ①`AskUserQuestion` 直前 ②長いサブタスク完了時 ③方針転換時。3〜5 行で「スコープ / 直近の進捗 / 残作業・次の一手」を含める。

## Issue / PR / ADR ドラフトはサマリレベルで提示

ターミナルに長文 Markdown 本文をベタ書きしない。提示は「タイトル案 / 属性 / 目的 1-2 行 / AC 見出しレベル / スコープ境界 / 関連 Issue」まで。本文ドラフトは構成 OK 後に生成し `gh issue create` 等へ渡す。

## ターミナル幅は測らず、崩れにくい形で出力する

Claude の bash は実端末に接続されておらず描画幅を計測できない。最初から崩れにくく書く: 比較表は 3 列まで・各セルは短い語句のみ / 長い説明は表の外の箇条書きに出す / ワイドテーブル・罫線アートを使わない。

## 不満サイン検知時のリカバリ

「おちょくって」「同じ話」「意味がわからない」「論点を外している」「長い」「冗長」「いえ、ちがう」等の不満サインを検知したら、論点を外している前提でリセットする: ①即謝罪（言い訳しない）②問いを 1 文で再言語化 ③「合っている」なら結論 1〜2 文で再回答 ④ズレた経緯を 1 文添える。

## 実装語彙はユーザーの語彙に翻訳する

AWS 用語・設計パターン名・フレームワーク用語を説明の主役にしない。まず一般語、必要なら後から「技術用語で〇〇と呼ぶ」と補足する。例: 2-Phase Commit →「全部揃ってから一括確定する仕組み」。

## やさしい日本語（文の組み立て・R1〜R8）

日本語出力の文単位の規律。技術用語のカタカナ語・コード識別子・コミットメッセージの英語は対象外。

- **R1 一文を短く**: 論点を詰め込まない。理由・条件は別文に切り出す。❌「〜であり、〜のため、〜だが」の複文多用
- **R2 主語・動作主を明示し能動態**: 誰が何をするかを落とさない。「それ / これ」の指示対象を直前に置く
- **R3 3 つ以上の並列は箇条書き**: 句読点で羅列しない
- **R4 一般語の外来語は日本語に**（技術用語は除外）: 例 エビデンス→証跡 / アサイン→割り当て。デプロイ / コミット等の技術語はそのまま
- **R5 二重否定を使わない**: ❌「対応していないことはない」→ ✅「対応しています」
- **R6 期限・数量は具体値**: ❌「後日」「いくつか」→ ✅ 日時（YYYY-MM-DD HH:MM）・数値・範囲「X〜Y」
- **R7 名詞句連結より動詞句**: ❌「PdM ヒアリング」→ ✅「PdM にヒアリングする」。動作名詞（ヒアリング / レビュー / 通読 / 把握 / 整備 / 分析 / 共有 / 承認 / 確認 / 反映）を名詞のまま連結しない
- **R8 project 固有名詞は役割を併記**: ファイル名 / テーブル名等は初出で役割を添える。❌「`legacy-analysis` を読む」→ ✅「現サイト機能の分析資料（`legacy-analysis`）を読む」。一般語・技術用語・ユビキタス言語・ディレクトリ名は除外
