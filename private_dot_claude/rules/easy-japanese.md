# やさしい日本語ルール

Claude → ユーザへの日本語出力 (チャット応答 / Issue / PR / コメント / 説明文)
で適用する **文の組み立て規律**。出典: <https://hellouniweb.com/columns/easy-japanese/>。
ネイティブ向けにスコープを絞って記事の一部ルールを採用しつつ、運用で出てきた
追加規律 (R7 名詞句→動詞句 / R8 project 固有名詞は役割併記) を含めて R1-R8 まで拡張している。

## 適用範囲

- Claude が日本語で出力する全テキスト (チャット応答 / GitHub Issue / PR 本文 /
  コメント / プラン文 / 報告サマリ等)
- **技術用語のカタカナ専門語は対象外** (デプロイ / リファクタリング /
  コンポーネント / コミット 等は保持する)
- コード識別子・関数名・コミットメッセージの英語部分は対象外

## スコープ外 (採用しなかった記事ルール)

- **「重要情報から伝える (結論ファースト)」**: `tameny-base:explain-discipline`
  スキルが既に強制しているため重複させない
- **「ふりがなを付ける」**: ネイティブ向けには過剰

## ルール

### R1: 一文を短くする

**原則**: 一文に複数の論点を詰め込まない。理由・条件は別文に切り出す。

- **Do**: 主語と述語が 1 対 1 の単文を基本にする / 「。」で区切って 2 文に分ける
- **Don't**: 「〜であり、〜のため、〜だが、〜となる」のような接続を重ねた複文を多用する

**言い換え例**:

- 通常: 「申請内容に不明な点がある場合は確認のため担当者から連絡することがあります」
- やさしい: 「申請内容が分からないときは、担当者から連絡します。そのときは確認に協力してください」

### R2: 主語と動作主を明示し、能動態を基本にする

**原則**: 誰が何をするのかが文から落ちないようにする。

- **Do**: 主語を明示する / 能動態で書く / 「それ」「あれ」「これ」が指す対象を直前に置く
- **Don't**: 主語を省略して文脈依存にする / 受け身を多用する / 行為者を曖昧にする

**言い換え例**:

- 通常: 「必要書類は窓口で確認されたあと返却されます」
- やさしい: 「窓口の担当者が必要書類を確認します。確認したあと、担当者が書類を返します」

### R3: 3 つ以上並べるものは箇条書きにする

**原則**: 並列情報は文章で列挙せず、箇条書きで視覚的に整理する。

- **Do**: 手順・条件・選択肢・持ち物リストは箇条書きにする / 並列要素は 1 行 1 項目で並べる
- **Don't**: 「A と B と C と D を持ってきてください」のように句読点で並べる / 箇条書きと長文を混在させる

**言い換え例**:

- 通常: 「申請のときは、本人確認書類、申請書、必要書類のコピー、手数料を持ってきてください」
- やさしい:
  > 申請のときは、次のものを持ってきてください。
  > - 本人確認書類
  > - 申請書
  > - 必要書類のコピー
  > - 手数料

### R4: 一般語の安易な外来語は日本語に言い換える (技術用語は除外)

**原則**: 日本語で十分通じる一般語の外来語は和語・漢語に言い換える。技術専門用語は対象外。

- **Do**: 一般語の外来語 (エビデンス / アサイン / コミット (約束の意) / オーソライズ 等) を
  日本語に言い換える / 業務文脈で誤解を生む外来語を避ける
- **Don't**: デプロイ / リファクタリング / コンポーネント / コミット (git の意) /
  リポジトリ 等の技術専門用語まで言い換える / 「バス」「ガス」「テレビ」のような
  定着した一般語を変える

**言い換え例**:

- 通常: 「エビデンスをアサインしてオーソライズしてください」
- やさしい: 「証跡を担当者に割り当てて承認してください」
- 技術文脈 (そのまま OK): 「main ブランチに直接コミットしないでください」

### R5: 二重否定を使わない

**原則**: 「〜ないことはない」「〜なくはない」のような否定の重なりを避ける。
肯定形か、単純な否定形のどちらかで書く。

- **Do**: 肯定形に組み替える / 単純な否定 1 段で表現する
- **Don't**: 「禁止していないわけではない」「不可能ではない」のような重複を使う

**言い換え例**:

- 通常: 「在留カード以外は必要ありません」
- やさしい: 「在留カードを持ってきてください」

- 通常: 「対応していないことはありません」
- やさしい: 「対応しています」 または 「対応します」

### R6: あいまいな表現を減らす (期限・数量は具体値で)

**原則**: 受け取り方が人によって変わる相対表現を避け、数値・範囲・固有値で書く。

- **Do**: 期限は日時 (YYYY-MM-DD HH:MM) で / 数量は数値で / 範囲は「X〜Y」で
- **Don't**: 「後日」「しばらく」「順次」「なるべく」「〜ごろ」「いくつか」のような
  相対表現で濁す

**言い換え例**:

- 通常: 「担当者が午前 11 時ごろに連絡します」
- やさしい: 「担当者が午前 10:30〜11:30 の間に連絡します」

- 通常: 「いくつかのファイルを修正しました」
- やさしい: 「3 ファイル (src/foo.ts / src/bar.ts / test/baz.test.ts) を修正しました」

### R7: 名詞句連結より動詞句で書く

**原則**: 「XX ヒアリング」「YY 通読」「ZZ 整備」のような **動作名詞 + 修飾名詞** の連結を避け、
動詞句に開いて「誰が何をどうする」を明示する。

- **Do**: 「PdM にヒアリングして要件を確認する」「legacy-analysis を読んで不足分を補う」
- **Don't**: 「PdM ヒアリング」「legacy-analysis 通読」「機能一覧整備」

**Why**: 動作名詞 (ヒアリング / 通読 / 整備 / 把握 / 分析 / 共有 / 確認 / 承認 / レビュー 等) を
名詞のまま並べると、「動作主 (誰が)」と「対象 (何を)」と「動作 (どうする)」が圧縮されて、
読み手が文ごとに頭の中で開き直す負担が発生する。特に `+` や `/` で 2 個以上連結すると
読みづらさが累積する。動詞句に開くと一読で意味が取れる。

**言い換え例**:

- 通常: 「現サイト操作 + legacy-analysis 通読 + PdM/BE ヒアリングを通じて整備する」
- やさしい: 「現サイトを操作し、legacy-analysis を読み、PdM と BE にヒアリングして整備する」

- 通常: 「Home カンプのステークホルダーレビュー後に展開」
- やさしい: 「Home カンプをステークホルダーがレビューし、承認されてから展開する」

**よくある名詞句 → 動詞句の置き換え表**:

| 名詞句 | 動詞句 |
|---|---|
| XX ヒアリング | XX にヒアリングする / XX に聞く |
| XX レビュー | XX をレビューする / XX を見直す |
| XX 通読 | XX を読む / XX に目を通す |
| XX 把握 | XX を把握する / XX を理解する |
| XX 整備 | XX を整備する / XX を揃える |
| XX 分析 | XX を分析する |
| XX 共有 | XX を共有する |
| XX 承認 | XX を承認する / XX の承認を得る |
| XX 確認 | XX を確認する |
| XX 制作 | XX を制作する |
| XX 反映 | XX を反映する |

**スコープ外**:

- **技術用語のカタカナ専門語**: 「デプロイ」「コミット」「リファクタリング」など、名詞のまま使う
  慣習が定着している技術語は対象外
- **慣用句として固まっている名詞句**: 「リアルタイム配信」「定期実行」「全体調整」など、
  動詞に開くと逆に読みづらい表現は OK
- **箇条書きの項目見出しとして使う場面**: 表のセルや箇条書きの先頭で名詞のまま使うのは可
  (例: 「## 分析」「対象: 機能整備」のような短い名詞句)
- **タイトル / 見出し**: 短く名詞句で締めるのは OK

### R8: project 固有名詞は「役割の説明」と併記する

**原則**: ファイル名 / ディレクトリ名 / ドキュメント名 / テーブル名 / ツール名など、
**一般語 / 技術用語 / ユビキタス言語ではない project 固有名詞** をテキスト中に出すときは、
その名称が **何の役割を持つか** を説明する語と併記する。

- **Do**: 「現サイト機能の分析資料 (`legacy-analysis`) を読む」 / 「予約データを保存する `bookings`
  テーブルを参照する」
- **Don't**: 「`legacy-analysis` を読む」 / 「`bookings` を見る」

**Why**: 名称だけだと、その project に新規参入する人や、別チームのレビュアーが「これは何のことか」
を別タブで開いて調べないと意味が取れない。役割の説明を併記すれば、名称を知らない読み手でも
文中で何の話か掴める。

**スコープ外**:

- **一般語**: API / DB / 関数 / クラス / Repository など
- **技術用語**: React / TypeScript / Figma / Slack / Notion など
- **ユビキタス言語 (project 共通言語)**: 「マイページ」「お見合い」「会員」など、project の DDD
  文脈で確立された用語 (project 内で短い説明なしで通じる語彙)
- **同じ body 内で 2 回目以降の参照**: 初出で説明したら、以降は名称のみで可。GitHub Issue / Task
  などはそれぞれ独立した body として扱う (= 各 Issue body 内で初出時に説明併記する)
- **ディレクトリ名 (末尾 `/`)**: `research/` / `docs/replacement/` など、複数ファイルを束ねる階層は
  ディレクトリ階層自体が役割の手がかりになるので説明不要
- **構造用語と組み合わさる固有名**: Figma の `WireFrame_マイページ` / `Design_マイページ` 等、
  Page / Section / Frame などの構造用語と組み合わせれば意味が取れる場合は説明不要

**判定の目安**:

- 「これは英単語の組み合わせを頭の中で訳さないと意味が取れないか?」と自問する
  - Yes → 説明併記
  - No (= 一般語 / 技術用語 / ユビキタス言語 / ディレクトリ / 構造用語との組み合わせ) → 名称のみ可
- **個別ファイル名 (.md / .ts / .py 等の拡張子付き) は原則すべて説明併記**。
  「英単語の組み合わせで役割が読み取れる」(`target-persona.md` 等) も、英単語の読み合わせは
  読み手の負担になるので併記対象とする
- ユビキタス言語かどうかの判定は project ごとに異なる。**迷ったら説明併記** が安全側

**言い換え例**:

- 通常: 「`legacy-analysis` を読む」
- やさしい: 「現サイト機能の分析資料 (`legacy-analysis`) を読む」

- 通常: 「`target-persona.md` にターゲット属性が記載されている」
- やさしい: 「`target-persona.md`（ターゲット属性）にターゲット属性が記載されている」
  - 英単語の組み合わせで役割が読み取れるファイルでも、読み手の脳内翻訳の負担を減らすため併記する

- 通常: 「`mypage-bookings` テーブルを参照する」
- やさしい: 「予約データを保存する `mypage-bookings` テーブルを参照する」

- 通常: 「`research/` 配下を整理する」
- やさしい: ディレクトリ名はスコープ外。そのままで可

**併記スタイルの推奨**:

ファイル名は **「`name.md`（役割）」(全角括弧で後置)** スタイルを基本とする。文の主役を名称に
おきたい場面では「役割 (`name.md`)」(半角括弧で並記) も可。同じ body 内で初出 → 2 回目以降は
名称のみで OK。

## このルール自体を Lint する観点

自分の出力を読み返すとき、以下を点検する:

- [ ] 一文に論点が 2 つ以上入っていないか (R1)
- [ ] 主語が省略されて「誰が」が分からなくなっていないか (R2)
- [ ] 3 つ以上の並列が句読点で羅列されていないか (R3)
- [ ] 技術用語以外で安易な外来語を使っていないか (R4)
- [ ] 二重否定で読み手に組み立て直しを強いていないか (R5)
- [ ] 期限・数量があいまいなまま放置されていないか (R6)
- [ ] 動作名詞 (ヒアリング / レビュー / 通読 / 把握 / 整備 / 分析 / 共有 / 承認 / 確認 / 制作 / 反映)
      を名詞のまま連結していないか / 動詞句に開いて動作主と対象を明示しているか (R7)
- [ ] project 固有名詞 (ファイル名 / ディレクトリ名 / ドキュメント名 / テーブル名等) の初出時に、
      役割の説明を併記しているか (R8) / 一般語・技術用語・ユビキタス言語・名前から自明なものは
      除外できているか
