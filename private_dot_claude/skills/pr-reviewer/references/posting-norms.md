# 投稿規範・判定の型・要約報告の指針

pr-reviewer が PR にレビューを投稿するときの文面規範、判定の使い分け、要約報告の書き方、投稿レシピをまとめる。

## 投稿規範（文面の書き方）

他者の PR に見えるコメントを書くときの原則。ユーザー（レビュー依頼された側）の実運用フィードバックから確立した。

1. **他レビュアーの既出指摘とその派生は除く**。人間レビュアーが既に指摘・解消した観点は投稿から外し、自分（L1 Claude + L2 Codex）が独自に見つけた観点に絞る。既出の言い換え・派生も除く。重複指摘はノイズになり判断コストを上げる。CodeRabbit（bot）の既出とは棲み分け、重複する場合は「賛同（refer）」に留めて二重化しない。
2. **感謝の枕詞を使わない**。「レビューありがとうございます」等は、自分がレビューを依頼された側（reviewer）の立場では逆になる。客観的な導入（例: 「〜を実際のコードと突き合わせて確認しました」）で始める。
3. **やさしい日本語で噛み砕く**。「文書にはこう書いてあるが、実物はこうなっている、だからこう直したい」の形にする。専門用語・フォーマット構造を先に出さない。
4. **番号や記号だけで語らない**。行番号・ID・略号は「何の箇所か」を言葉で添える（例: ✗「54 行」→ ◯「検証ギャップの節（54 行付近）」）。トップコメントには生の行番号を極力出さず、行コメント側でアンカーする。
5. **心理的安全性**: 冒頭で良い点（praise）を簡潔に述べる。ただし既出の praise とは重複させない。
6. **判定はマージを止める意図かどうかを明示**する（「これはマージを止める指摘ではなく、修正のお願いです」等）。

## 判定の型（comment / approve / request-changes）

判定は指摘件数だけで機械的に決めず、**対象文書の性質**と**この PR の approve が何をトリガーするか**を踏まえる。

| 状況 | 判定 | 書きぶり |
|---|---|---|
| should-fix のみ・must-fix なし | comment | 修正希望を伝え、作成者判断に委ねる。blocking しない |
| notes/ ドラフト（status: 提案中） | approve + 指摘 | 「ドラフト追加自体は承認。`framework/` へ正本化する際に直して」 |
| 実際に使う正本文書 | approve + 条件付き指摘 | 「承認。ただし下記を直したうえでマージして」 |
| must-fix があるが対象はドラフト | 要相談 | approve + 指摘か request-changes か、ユーザーに判断を仰ぐ |
| must-fix が設計の根本を揺るがす / 明確な security・本番影響 | request-changes | 躊躇せずブロック |

補足:
- **この PR の approve が「正典化・昇格」のトリガー**になっている場合（PR 本文に「承認後 framework/ へ昇格」等）、誤りを含んだまま approve すると誤りが正典に伝播する。must-fix が残るなら request-changes も筋が通る。最終的な強度はユーザーに確認する。
- 日本語チームでは request-changes が強く受け取られやすい。明確な blocker でなければ approve + 指摘 / comment を選ぶ方が心理的安全性を保てる。ただし対象がドラフトか正本か、approve が何を意味するかで変わるので、**型に当てはめる前にユーザーと握る**。

## 要約報告の指針（Step 4 の報告）

ユーザーが全 diff を読まずに判定できるよう、各 PR を**概要レベル**に噛み砕く。含める要素:

- **経緯**: なぜこの PR が生まれたか（きっかけの Issue / 上流の決定）。日常語で。
- **目的・内容**: 何を変えるか。変更ファイルの羅列でなく「何のための変更か」。
- **状態の要点**: 他レビュアーの状況、変更要求が解消済みか、CI / CodeRabbit。
- **判定に効く論点**: ユーザーの判断を左右する点（例: 「実物にないファイルが記載されている」「承認すると正典化が進む」）を先に出す。

番号・略号の羅列を避け、「それが何を指すか」を言葉で添える。

### 報告テンプレート（1 PR あたり）

以下の型で 1 PR を報告すると、ユーザーが概要だけで判定できる。ユーザーの判断コストを下げることがこのスキルの主目的なので、この型を崩さない。

```text
**#<番号> <タイトルを一言で>** — 推奨判定（理由の核を一言）
- 経緯: なぜこの PR が生まれたか。日常語で 1 文
- 内容: 何を変えるか。ファイル羅列でなく「何のための変更か」で 1〜2 文
- 状態: 他レビュー・変更要求の解消状況・CI / CodeRabbit を 1 行
- 判定に効く論点: ユーザーの判断を左右する点（あれば。なければ省略）
```

複数 PR のときは、まずこの型で全 PR を一覧し、そのあと 1 PR ずつ投稿ドラフトと判断を仰ぐ（Step 4）。

## 投稿レシピ（Pull Request Reviews API）

`gh pr review` は行コメント非対応なので、Reviews API に POST する。日本語をコマンド文字列に埋めず、**ファイル + `jq --rawfile`** で組み立てる（tool-call-hygiene 準拠）。

```bash
# 1. 文面をファイルに書き出す（Write ツールで）
#    /tmp/pr<N>-top.md（トップコメント）, /tmp/pr<N>-c1.md, c2.md, ...（行コメント）

# 2. 行位置を対象ブランチの worktree で確定（diff の追加行=RIGHT にのみ付けられる）
grep -n "<該当行の一意な文字列>" <worktree>/<path>

# 3. jq --rawfile で JSON 組み立て
P="<repo-relative path>"
jq -n --rawfile body /tmp/pr<N>-top.md \
      --rawfile c1 /tmp/pr<N>-c1.md \
      --rawfile c2 /tmp/pr<N>-c2.md \
      --arg p "$P" \
  '{event:"COMMENT", body:$body, comments:[
     {path:$p, line:<L1>, side:"RIGHT", body:$c1},
     {path:$p, line:<L2>, side:"RIGHT", body:$c2}
   ]}' > /tmp/pr<N>-review.json
jq -e . /tmp/pr<N>-review.json >/dev/null  # valid 確認

# 4. 投稿（event は COMMENT / APPROVE / REQUEST_CHANGES）
gh api --method POST /repos/<owner>/<repo>/pulls/<N>/reviews \
  --input /tmp/pr<N>-review.json --jq '{id, state, html_url}'
```

注意:
- `line` は diff に含まれる行（追加行 or 文脈行）でないと API が 422 を返す。新規追加節・新規ファイルは全行が対象。
- 複数ファイルにまたがる行コメントは `comments` 配列で各 `path` を指定する。
- 行指定できない指摘（PR 本文全体の方針・PR 間整合など）は**トップコメントに直接**書く。
- 分類ラベルは文脈に合わせて日本語化してよい（例: notes ドラフトなら `[正本化までに要修正]`、正本なら `[should-fix]`）。approve と `must-fix`（本 PR 内で対応の意）が矛盾しないよう、ラベルを状況に合わせる。
