# ADR Review Checklist (5 軸)

`/review-adr` スキルが L1 Claude レビューで使用する評価軸。各軸の `findings` を列挙する際の判断基準。

## 軸 L: 論理一貫性

ADR の各セクション間の論理連鎖が崩れていないかを確認する。

### チェック項目

- [ ] **コンテキスト → 判断論点の連鎖**: コンテキストで提起された問題が「判断論点」に網羅されているか
- [ ] **判断論点 → 検討選択肢**: 各論点に対し選択肢が言及されているか
- [ ] **検討選択肢 → 決定**: 採用された選択肢が「検討した選択肢」セクションに明記されているか
- [ ] **決定 → 理由**: 決定の各要素に対応する理由が「理由」セクションにあるか
- [ ] **決定 → 影響**: 決定が引き起こす影響が「影響」セクションにあるか
- [ ] **コンテキストの参照**: コンテキストで「ADR-XXX で確立した X を踏襲」と書いた場合、決定 / 理由でその踏襲が具体的に現れているか

### 典型的な findings

- 「コンテキストで論点 X を提起したが、決定で言及がない」
- 「採用した選択肢の理由が、別の選択肢の理由と重複している（差別化の根拠が弱い）」
- 「影響セクションが利用組織への影響のみで、開発フローや運用への影響が抜けている」

## 軸 C: 既存 ADR との矛盾

参照する既存 ADR との整合性、上書き / 更新時の明示性を確認する。

### チェック項目

- [ ] **参照 ADR の方針との矛盾**: 「ADR-XXX を踏襲」と書きつつ、実際は別方針になっていないか
- [ ] **上書き / 非推奨化の明示**: 既存 ADR を上書きする場合、対象 ADR のステータスを「非推奨」化するか後続 ADR で明示しているか
- [ ] **参照 ADR 番号の整合**: ADR-XXX という参照番号が実在するか、リンクが正しいか
- [ ] **同一論点の重複扱い**: 既に別 ADR で扱われている論点を再決定していないか（している場合、その理由が明示されているか）

### 典型的な findings

- 「ADR-014 を踏襲と書いているが、判断基準の管理層が ADR-014 と異なる」
- 「ADR-XXX を上書きする決定だが、ADR-XXX のステータス変更について言及がない」
- 「参照されている ADR-099 が実在しない」

## 軸 G: 抜け漏れ

ADR として必要な要素が揃っているか、スコープ管理が適切かを確認する。

### チェック項目

- [ ] **スコープ外の明示**: 本 ADR で扱わない項目が「スコープ外」として明示されているか
- [ ] **検討選択肢 ≥ 2**: 最低 2 つの選択肢が比較されているか（採用案のみは ADR でなく仕様書）
- [ ] **ネガティブ影響の記載**: 採用案のデメリット / 既存への影響がリストアップされているか
- [ ] **関連 Issue / ADR リンク**: 親 Issue / 元 Issue / 関連 ADR が明示されているか
- [ ] **判断の根拠データ**: コスト / 時間 / 性能 などの定量比較が必要な場合、根拠が示されているか
- [ ] **後続作業の明示**: 雛形配置や別 Issue で対応する項目がリストアップされているか

### 典型的な findings

- 「『投資レベル別構成』を採用しているが、L0 のみ採用するデメリットの記載がない」
- 「『スコープ外』セクションがなく、本 ADR の範囲が曖昧」
- 「比較対象が 1 つ（採用案）のみで、なぜ他案を採用しないかの判断材料がない」

## 軸 A: 軸選定の妥当性

ADR の中心的な比較軸（投資レベル / 階層モデル / 二層化等）の選定根拠を確認する。

### チェック項目

- [ ] **軸の選定理由**: なぜその軸（例: 投資レベル / 観測対象階層）を選んだかが明示されているか
- [ ] **軸と決定の整合**: 軸で分類した結果が決定セクションの構成と一致しているか
- [ ] **軸の網羅性**: 重要な分類が軸で取りこぼされていないか
- [ ] **軸の独立性**: 二次元マトリクスを使う場合、軸同士が直交しているか（重複していないか）
- [ ] **代替軸の検討**: 別の軸で分類する選択肢が「検討した選択肢」に挙がっているか

### 典型的な findings

- 「観測対象階層 × 4 Golden Signals マトリクスを採用しているが、なぜこの 2 軸かの根拠が薄い（Well-Architected の OPS04 等の参照根拠を補強推奨）」
- 「投資レベル軸を採用しているが、投資レベル間の境界（L0 と L1 の差）が定量的に示されていない」
- 「『テンプレート構造 / プロジェクト属性』二層化を踏襲しているが、本 ADR の各項目がどちらに属すかの分類が一覧化されていない」

## 軸 N: 表記・整合（軽量）

CodeRabbit と重複しがちなため、初出のみ指摘する。

### チェック項目

- [ ] ADR 番号の整合（ファイル名と本文の番号が一致）
- [ ] 見出し階層（## / ### の使い分け）
- [ ] リンクの存在性（GitHub Issue / 外部 URL のリンク切れ）
- [ ] 用語の一貫性（同一概念に複数の語を使い分けていないか）
- [ ] Markdown フォーマットエラー（テーブル崩れ等）

### 典型的な findings（初出のみ指摘）

- 「ファイル名は `016-observability.md` だが本文は `ADR-0016`、ゼロ埋め桁数の整合確認推奨」
- 「『観測指標』『監視指標』『メトリクス』が混在、用語統一を推奨」

## severity 判定基準

| severity | 基準 |
|----------|------|
| `must-fix` | 論理矛盾 / 既存 ADR との整合崩れ / 必須セクション欠落（コンテキスト・決定・理由・影響） |
| `should-fix` | 抜け漏れ / 軸選定根拠の弱さ / 検討選択肢の不足（採用案のみ） |
| `nit` | 表記ゆれ / 軽微な抜け漏れ / 改善提案 |
