# Work Issue Skill

GitHub Issue 番号を引数に受け取り、その Issue を **着手 → AC 判断 → 修正 → セルフレビュー → PR 作成 → CodeRabbit 対応 → Approved/CI 確認 → 動作確認 → マージ指示伺い** まで完遂させるための `/goal` コマンド文字列を生成して提示する。

## このスキルの役割（重要）

**このスキルは実作業をしない**。Issue 番号を埋めた `/goal` ワンライナーを生成し、ユーザーが貼り付けて実行するための文字列を提示するだけ。

理由: `/goal` はビルトインのスラッシュコマンドで、セッションスコープの Stop hook（条件達成まで停止をブロックする強制力）を設置する。この Stop hook の強制力はユーザーが `/goal` を手入力したときにのみ有効になる。スキル・サブエージェント・バックグラウンド・hook のいずれからも `/goal` を起動できないことが Claude Code 公式仕様で確定している（スラッシュコマンドはツールではなく、hook は `/` コマンドをトリガーできず、サブエージェントの hook は親セッションに波及しない）。

したがって `/goal` の強制力を活かすには、文字列を生成 → ユーザーが貼って実行、という経路を取る。

## Procedure

### Step 1: Issue 番号の確定

- 引数から Issue 番号を取り出す。先頭の `#` は除去し、数値部分のみを使う（例: `#602` / `602` どちらも `602` として扱う）。
- 引数が無い・数値でない場合は、対象 Issue 番号をユーザーに尋ねてから先へ進む。

### Step 2: Issue の実在確認（任意・推奨）

対象が正しいか取り違えを防ぐため、Issue のタイトルと状態を 1 回だけ取得して提示する。失敗しても致命的ではない（番号だけで続行してよい）。

```bash
gh issue view <番号> --json number,title,state --jq '"#\(.number) [\(.state)] \(.title)"'
```

- 取得できたらタイトルを 1 行で示し、「この Issue で合っているか」を添える。
- CLOSED の場合はその旨を警告する（既に完了済みの可能性）。

### Step 3: `/goal` コマンド文字列の生成・提示

以下のテンプレートの `<番号>` を確定した Issue 番号で置換し、**そのまま貼り付け可能な 1 行**としてコードブロックで提示する。

```text
/goal Issue #<番号> を着手してください。Issue の AC が明確でない場合は判断を仰いでください。修正後はセルフレビューを行い、問題がなくなったら PR を作成、CodeRabbit のレビューを待ち必要に応じて review-pr スキルで指摘対応、CodeRabbit Approved・CI 通過を確認できたら動作確認のうえマージ指示を仰いでください。
```

提示時に「この行をプロンプトに貼って実行すると、Stop hook 付きで完遂まで自走します」と 1 行添える。

### Step 4: 終了

文字列を提示したらこのスキルの仕事は完了。ここで実作業（ブランチ作成・実装・PR）には**入らない**。実作業はユーザーが `/goal` を貼って実行したターンから始まる。

## ワークフローの中身（生成される /goal が辿る流れ）

`/goal` 実行後に Claude が辿る想定フロー。スキル利用者の理解のために記す（このスキル自体は実行しない）。

| 段階 | 内容 | 関連スキル/ツール |
|------|------|------------------|
| 着手 | Issue 本文を読み AC・スコープを把握。AC が不明確なら判断を仰ぐ | `gh issue view` |
| 修正 | 委譲先を判定して実装（既定は Codex 委譲） | task-delegation / codex:rescue |
| セルフレビュー | 問題がなくなるまで確認 | self-review |
| PR 作成 | ブランチ作成 → コミット → push → PR | branch / commit / pr |
| レビュー対応 | CodeRabbit のレビューを待ち、指摘があれば対応 | review-pr |
| マージ前確認 | CodeRabbit Approved・CI 通過・動作確認 | — |
| マージ | **ユーザーの明示指示を仰いでから実行**（git-safety 準拠） | — |

## 制約

- **実作業を勝手に始めない**。このスキルは `/goal` 文字列の生成・提示までで止まる（実行方式の設計決定: テキスト生成のみ）。
- 生成する `/goal` 文には Issue 番号を必ず埋める。番号未確定のまま提示しない。
- マージはワークフロー上も「指示を仰ぐ」で止める設計。`/goal` 文に自動マージを含めない（git-safety.md 準拠）。
