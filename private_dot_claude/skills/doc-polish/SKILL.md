# Skill: doc-polish

## 概要

Markdown ドキュメント（計画文書・ADR・ガイド等）の日本語表現を研磨し、改善を適用する。レビュー観点は 4 つ:

1. **一般的でない表現**: 社内・業界で通じない独特な言い回し（例:「〜に接地する」「波状起票」「tail 収束」「並行充填」）
2. **難しい言い回し**: 長文・主語省略・二重否定・動作名詞の連結（`~/.claude/rules/communication-style.md` の R1〜R8 が正本）
3. **造語・略語・記号参照**: 独自略語の展開、ID・節番号参照への名称併記、独自グルーピング語の定義（下記「表記規律」が正本。初出に説明を添えるだけでは不十分）
4. **図の不足**: 文章・表だけでは構造や時系列が掴みにくい節への Mermaid 図挿入（`~/.claude/docs/mermaid-conventions.md` が正本）

役割分担: **編集の実行は Codex（T2）へ委譲**し、**司令塔（T1）は diff 検証・事実照合・構文検証・コミット判断**を担う（`task-delegation` スキルの体系に従う）。ただし日本語の表現レベルの修正は司令塔が直接 Edit してよい（feedback メモリ「日本語の表現修正は Sonnet に委譲しない」）。

## 表記規律（記号・略語を単独で歩かせない）

2026-07-15 の PR #1297（スケジュール・ゲート計画文書）へのユーザーフィードバックが起点。番号・略語は書き手の索引にすぎず、読み手の理解を運ばない。観点 3 の具体規範として、以下を機械的に適用する。

| # | 規範 | Don't | Do |
|---|---|---|---|
| A | 工数単位は日本語の慣用単位で書く | `415 PD` / `18 PD/月` | `415 人日` / `18 人日/月`（人月も同様。参照元が PD 表記でも本文は人日に統一し、凡例でその旨を注記） |
| B | 節番号参照は節タイトルを併記する（またはアンカーリンク） | `（§2.4）を参照` | `実現可能性の評価（§2.4）を参照` |
| C | 工程・職能の独自略称は展開する | `BD+DD` / `SI` / `FI` | `基本設計＋詳細設計` / `サーバー実装` / `フロント実装`。略すなら FE / BE / Infra / Native のような一般に通じる職能略称のみ |
| D | 文書内 ID は定義表以外で名称を併記する | `M5 の律速` / `CP-2 までに` / `G4 を通過` | `M5 詳細設計・実装の律速` / `CP-2（MVP 第一回段階判断）までに` / `G4 実装完了ゲートを通過` |
| E | 独自のグルーピング語は凡例で定義し出典を添える | `必須13` / `フル20カテゴリ` | 凡例で「MVP 区分（`mvp-scope.md`）の必須 13 カテゴリ」と定義したうえで `必須 13 カテゴリ` と書く。可能なら Milestone / Epic / Issue など GitHub 上の単位・粒度で表現する |
| F | **Mermaid 図・gantt のラベルも本文と同じ規律** | flowchart のラベル `W0/W1 基本設計確定` / `コア(C1-C4)` | `認証・会員情報・写真イメージの基本設計確定` / `コアジャーニー（C1 入会・初期設定 / …）`。図は本文から切り離されて単体で読まれるため、本文で定義済みでも図中では名称を書く |
| G | 役割・プロセス略語も初出で展開する | `SSOT` / `DoD` / `DoR` / `AC` / `IA` / `WIP` / `TL` / `PL` / `WF` 単独 | `SSOT（Single Source of Truth＝単一の正本）` / `完了定義（DoD）` / `着手条件（DoR）` / `受け入れ基準（AC）` / `情報アーキテクチャ（IA）` / `WIP 制限（同時進行数の上限）` / `テックリード` / `PL＝プロジェクトリーダー` / `ワイヤーフレーム（WF）` |
| H | 同一略語の多義使用を禁止・解消する | `SP`（ストーリーポイント）と `SP 主体`（スマホ）と `SP-1`（同期点）の混在、`P`（ページ数）と `P`（先行度スコア） | 意味ごとに別表記へ分離し（`スマホ画面主体` / `24 ページ`）、残る略語は凡例で「どの意味か」を宣言する |

適用の例外:

- 定義表・凡例・一覧表など「その ID を定義している場所」では ID 単独で書いてよい（例: Milestone 一覧表の M1〜M9）
- 外部文書が定義する識別子（`BD-BE-4` / `BE-S{step}` 等）は変更せず、初出で役割を併記する
- Mermaid のノード ID は ASCII のまま（規律 F の対象は表示ラベルのみ）

### 網羅チェックの手順（規律の適用漏れを防ぐ）

1 回の通読で拾おうとしない。**機械的な棚卸し → 修正 → 独立レビュアーのダブルチェック**の 3 段で回す:

1. **棚卸し**: `W[0-9]` / `C[0-9]` / `CP-[0-9]` / `G[0-9]` / `M[0-9]` / 大文字 2〜4 字の略語等のパターンで grep し、**Mermaid ブロック内か否かを区別して**全出現をリスト化する（2026-07-15 の失敗: 本文だけ直して Mermaid ラベルを見逃し、ユーザーから「全然治ってない」と再指摘を受けた）
2. **修正**: リストの全項目に規律 A〜H を適用する
3. **ダブルチェック（必須）**: 修正後に Codex（`codex:codex-rescue`・read-only レビュー指示）へ「初見の読み手が解読できない表記の列挙」を委譲し、指摘を突き合わせる。Codex 未導入環境では T3 実行サブエージェントで代替する

### `review-doc` との棲み分け

| スキル | 動作 | 使いどき |
|---|---|---|
| `review-doc` | 指摘の提案のみ。ファイルは編集しない | レビュー結果だけ欲しいとき |
| `doc-polish`（本スキル） | 改善を適用しコミットまで行う | 「直して」まで依頼されたとき |

## 実行手順

### Step 1: 対象の展開と現状把握

- PR 番号指定なら `gh pr view <N> --json files,headRefName,body` で変更ファイルとブランチを特定する
- ブランチ切替が必要で、checkout が untracked ファイルと衝突する場合は **worktree に展開する**（`git worktree add <scratchpad>/pr<N> <branch>`）。main の作業ツリーを壊さない
- `grep -n '^#' <file>` で見出し構造、``grep -c '^```mermaid' <file>`` で図の有無を把握する（`grep -c 'mermaid'` は本文中の言及も数えるため使わない）。図が 0 件の長文文書は図挿入の有力候補

### Step 2: 委譲プロンプトの生成

[references/codex-prompt-template.md](references/codex-prompt-template.md) を雛形に、対象ファイル・作業ディレクトリ・文書固有の制約を埋めたプロンプトを **scratchpad にファイルとして書き出す**（コマンド文字列に日本語長文を埋めない。`tool-call-hygiene` 準拠）。

雛形の「変更してはならないもの」は毎回具体化する。特に:

- 他文書から相互参照される**見出し・節番号**
- 計画値・実測値などの**数値・日付**
- 文書固有のタグ運用（例: `[既定] / [提案] / [要判断]`）
- 相対リンク・Issue 参照番号

### Step 3: Codex へ委譲

`codex:rescue` 経由で委譲する（Bash 直叩き禁止）:

```text
/codex:rescue --model gpt-5.6-terra <対象の 1 行サマリ>。詳細指示は <プロンプトファイルの絶対パス> を読むこと。作業ディレクトリは <worktree の絶対パス>
```

800 行超・複数文書で難度が高い場合は `--model gpt-5.6-sol` に切り替える。Codex 未導入環境では T3 実行サブエージェント（重い実行役）へフォールバックする。

### Step 4: 司令塔の diff 検証（必須）

Codex の完了報告を鵜呑みにせず、以下を自分で確認する:

```bash
git diff --stat
git diff -U0 | grep -E '^[+-]#{1,4} '   # 見出し変更の検出（原則ゼロであること）
git diff                                  # 全文レビュー
```

チェックリスト:

- [ ] 見出し・節番号・タグ・リンク・Issue 番号が不変
- [ ] 数値・日付が不変（図に転記された数値は本文の該当節と突合する）
- [ ] **事実関係の言い換えは引用元と照合する**（例:「削除」→「クローズ」の言い換えは Decision Record 原文を確認して矛盾がないことを確かめる）
- [ ] 図のノード ID とラベルが食い違っていない（例: ID が `C` でラベルが「案 D」）
- [ ] 中国語字形の漢字が混入していない

軽微な最終調整（日本語表現・ノード ID 等）は司令塔が直接 Edit する（Sonnet へ再委譲しない）。

### Step 5: Mermaid 構文検証

図を追加・変更した場合は push 前に必須:

```bash
bash ~/.claude/skills/doc-polish/scripts/check-mermaid.sh <対象 md の絶対パス>
```

全ブロック `PASS` を確認する。`FAIL` が出たら該当図を修正して再実行する。

### Step 6: 衛生チェック → コミット → 後片付け

- `git diff --numstat | awk '$1 > 1000'` で大型変更、`git diff --name-only | grep -E '\.terraform/|\.DS_Store|node_modules/|dist/'` で artifact 混入を確認する
- コミット規約: 件名は英語 Conventional Commits（例: `docs: improve wording and add diagrams to <対象>`）+ Co-Authored-By トレイラー
- push 先は feature ブランチのみ（protected ブランチへの直接 push 禁止。`git-safety.md`）
- worktree を使った場合は `git worktree remove <path>` で後片付けする

### 完了報告

以下をまとめて報告する: 変更カテゴリ別の一覧（表現修正 / 略語説明 / 図追加 / 整理）、検証結果（見出し・数値不変、事実照合、Mermaid PASS）、コミットハッシュ。

## エラーハンドリング

| 事象 | 原因 | 対応 |
|---|---|---|
| checkout が untracked と衝突 | main に旧 tracked ファイルの残骸 | worktree に展開する（Step 1）。untracked を消さない |
| Codex が制約を破った（見出し・数値変更） | プロンプトの制約が曖昧 | 該当箇所を司令塔が Edit で戻すか、制約を具体化して再委譲 |
| Mermaid parse FAIL | 日本語ノード ID・構文誤り | `mermaid-conventions.md` の落とし穴節を参照して修正 |
| Codex 未導入 | プラグインなし | T3 実行サブエージェント（重い実行役）へフォールバック |
