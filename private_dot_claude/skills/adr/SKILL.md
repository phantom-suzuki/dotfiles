# ADR Creation Skill

技術的な意思決定を ADR として記録する。
複数の選択肢を比較検討した設計判断、既存アーキテクチャに影響する変更、
半年後に「なぜこうなっている？」と聞かれそうな判断が対象。

## Parameters

| パラメータ | デフォルト | 説明 |
|-----------|-----------|------|
| タイトル/説明 | なし | ADR のタイトルまたは決定内容の自然言語説明 |

## Step 0: プロジェクト検出

[references/directory-detection.md](references/directory-detection.md) のロジックに従い、以下を自動検出する。

1. **ADR ディレクトリ**の特定
2. **既存テンプレート**の有無（`template.md` の存在確認）
3. **命名規則**の検出（既存ファイル名パターンを分析）
4. **次の ADR 番号**の算出

検出できない場合（ADR ディレクトリが存在しない等）:
- git リポジトリのルートに `docs/adr/` を作成するか確認する
- git リポジトリでない場合はエラーを返し終了する

## Step 1: コンテキスト収集

ユーザーの入力と会話コンテキストから、ADR に必要な情報を収集する。

### 会話コンテキストからの自動抽出

直前の会話で設計判断が行われていた場合、以下を自動的に抽出する:
- 何が課題だったか（背景）
- どんな選択肢を検討したか
- 何を選び、なぜ選んだか
- トレードオフは何か

### 対話による補完

自動抽出で不足する場合、以下を確認する:

1. **背景と課題**: なぜこの判断が必要になったか
2. **判断要因**: ビジネス要件・技術的制約・運用要件
3. **検討した選択肢**: 2つ以上（比較対象がない場合は `/adr-note` を提案）
4. **決定**: どの選択肢を採用したか、理由
5. **影響**: ポジティブ・ネガティブ双方

全てを聞く必要はない。会話の流れで自然に収集し、不足分のみ確認する。

## Step 2: テンプレート選択

以下の優先順位でテンプレートを決定する:

1. **プロジェクト固有テンプレート**: Step 0 で検出した `template.md` があればそれを使用
2. **標準テンプレート**: [templates/adr-standard.md](templates/adr-standard.md)
3. **簡潔テンプレート**: 選択肢が1-2個で詳細な比較が不要な場合は [templates/adr-lightweight.md](templates/adr-lightweight.md)

プロジェクト固有テンプレートを使う場合でも、YAML frontmatter に以下のフィールドがなければ追加する:
- `status`
- `date`
- `category`
- `ai_instruction`

## Step 3: ADR 生成

テンプレートに基づきドラフトを作成する。

### YAML Frontmatter の自動生成

```yaml
---
status: proposed
date: {今日の日付 YYYY-MM-DD}
decision-makers: []
category: {内容から自動判定}
related: []
ai_instruction: "この ADR は {対象} に関する意思決定を記録しています。{参照タイミング}。"
---
```

**category の判定基準:**

| category | 対象 |
|----------|------|
| engineering | コード設計・アーキテクチャ・技術選定 |
| infrastructure | インフラ・デプロイ・CI/CD |
| process | 開発プロセス・ワークフロー |
| tools | ツール選定・設定 |
| security | セキュリティ・認証・認可 |

**ai_instruction の生成ルール:**
- 1-2文で ADR の用途と参照タイミングを明示する
- 「この ADR は {対象領域} に関する意思決定を記録しています。{どんな作業時に参照すべきか}。」

### セクション記述の原則

- **背景と課題**: 技術に詳しくない人でも「何が問題だったか」がわかる書き方にする
- **判断要因**: ビジネス要件・技術的制約・運用要件の3カテゴリで分類する
- **検討した選択肢**: 番号付きリストで列挙する
- **決定**: 採用した選択肢を太字で明示し、判断要因との対応を記述する
- **各選択肢の比較**: Good / Bad 形式で統一する
- **結果と影響**: ポジティブ・ネガティブは必須。ユーザー影響・運用影響・セキュリティは該当する場合のみ記載する（該当しないセクションは削除する）
- **検証方法**: 実装を伴う決定の場合のみ記載する
- **関連情報**: 設計書リンク、関連 ADR、再検討条件を記載する
- **変更履歴**: 初版作成の1行を記載する

## Step 4: レビュー

生成したドラフトをユーザーに提示する。

提示の形式:
- ファイルを Write で書き出す
- ファイルパスとともに内容の要約を提示する
- 「修正点があれば指示してください。なければこのまま進めます。」と確認する

フィードバックがあれば Edit で反映し、再提示する。

## Step 5: Supersede 処理

ユーザーが「既存の ADR-XXXX を置き換える」と指示した場合、または会話コンテキストから既存 ADR を置き換える判断と推測される場合:

**Supersede は破壊的操作のため、必ずユーザーに確認してから実行する。**
「ADR-XXXX を置き換えますか？」と AskUserQuestion で確認し、承諾を得てから以下を実行する。

### 旧 ADR の更新

1. 旧 ADR の frontmatter `status` を `superseded` に変更する
2. 旧 ADR の frontmatter に `superseded_by: ADR-{新番号}` を追加する
3. 旧 ADR のタイトル直後に以下の注意書きを挿入する:

```markdown
> **Note**: この ADR は [ADR-{新番号}](./{新ADRファイル名}) に置き換えられました。
```

### 新 ADR への参照

新 ADR の `関連情報` セクションに以下を追記する:

```markdown
* Supersedes: [ADR-{旧番号}](./{旧ADRファイル名})
```

新 ADR の frontmatter `related` にも旧 ADR 番号を追加する。

## Step 6: インデックス更新

### ADR 目次の更新

ADR ディレクトリ内に `README.md` や目次ファイルがあれば、新しい ADR のエントリを追加する。

### CLAUDE.md への ADR ディレクトリ参照

CLAUDE.md に ADR ディレクトリへの参照がなければ、以下の追記を提案する（初回のみ）:

```markdown
## 設計判断

- ADR: `docs/adr/`
- ステータスの扱い: accepted = 従う、proposed = 参考情報、deprecated/superseded = 無視
```

個別の ADR 要約を毎回追記する必要はない。AI は必要に応じて ADR ディレクトリを参照する。
