# ADR Note Creation Skill

正式な ADR ほど重い形式が不要な、気軽なナレッジ記録を作成する。
調査結果、手順メモ、設計の補足情報、トラブルシューティング記録など、
「後で参照したい」情報を素早くドキュメント化する。

**正式 ADR との使い分け:**
- 複数の選択肢を比較検討した → `/adr`
- 「なぜこうした？」と後で聞かれそうな設計判断 → `/adr`
- 調査結果・手順・ナレッジを残したい → **`/adr-note`（このスキル）**

## Parameters

| パラメータ | デフォルト | 説明 |
|-----------|-----------|------|
| タイトル/説明 | なし | ノートのタイトルまたは内容の自然言語説明 |

## Step 0: プロジェクト検出

ADR ディレクトリの検出ロジックは `/adr` スキルと共通（詳細は [../adr/references/directory-detection.md](../adr/references/directory-detection.md) を参照）。以下の順序で探索する:

1. `docs/adr/`, `docs/adrs/`, `docs/decisions/`, `docs/architecture/decisions/`, `adr/`, `adrs/`, `decisions/`
2. サブディレクトリ内の上記パターン（例: `engineering/adrs/`）

検索は git リポジトリのルートから行う。
ADR ディレクトリが存在しない場合は `docs/adr/` を作成するか確認する。
git リポジトリでない場合はエラーを返し終了する。

ノートのファイル名は正式 ADR の連番とは独立した **日付ベース** を採用する:

```
YYYY-MM-DD-{タイトルのスラッグ}.md
```

例: `2026-04-09-gemini-cli-rate-limits.md`

同日に複数作成する場合はサフィックスを付与する（`-2`, `-3`）。
正式 ADR の採番空間とは完全に分離されるため、混在の問題が生じない。

## Step 1: コンテキスト収集

会話コンテキストから自動的に以下を抽出する:
- 何を記録するか（タイトル）
- なぜ記録するか（背景）
- 記録すべき内容

引数でタイトルが指定されていれば、追加の対話なしに直接生成に進む。
会話の流れで記録すべき内容が明確な場合も、追加の質問は最小限にする。

## Step 2: ノート生成

[templates/note.md](templates/note.md) に基づきノートを生成する。

### YAML Frontmatter

```yaml
---
title: "{タイトル}"
last_updated: {今日の日付 YYYY-MM-DD}
status: draft
category: {内容から自動判定}
ai_instruction: "{AIがこのノートをどう活用すべきかの1文}"
---
```

**category の自動判定:**

| category | 対象 |
|----------|------|
| engineering | コード設計・技術調査・実装パターン |
| infrastructure | インフラ構成・デプロイ手順 |
| process | 開発プロセス・ワークフロー |
| tools | ツール設定・使い方 |
| troubleshooting | トラブルシューティング・障害対応 |
| research | 技術調査・PoC 結果 |

### セクション構成

- **背景**: なぜこのナレッジを残すのか（1-2文）
- **内容**: ナレッジ本体。形式は自由（箇条書き、表、コード例、手順リスト等）

内容が長くなる場合は適切にサブセクション（`###`）で分割する。

## Step 3: ファイル書き出し

1. ファイルを Write で書き出す
2. ADR ディレクトリ内に README.md や目次があれば、エントリを追加する
3. ファイルパスを提示して完了を通知する
