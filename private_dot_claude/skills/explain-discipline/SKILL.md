# Explain Discipline

技術的説明や設計議論において、論点を外した冗長回答・同義反復・周辺情報羅列・矛盾を防ぐための説明規律スキル。

## 発動トリガー

以下のいずれかに該当したら **必ず本スキルの手順を踏む**:

- ユーザーが「なぜ」「理由」「どうして」を含む問いを投げた
- ユーザーが「わからない」「意味がわからない」「同じ話」「論点を外している」「おちょくって」「確信」等の不満・困惑サインを発した
- 設計議論で複数の選択肢（A 案 / B 案 等）を提示する場面
- 技術的な仕組み・アーキテクチャの説明依頼
- ユーザーが質問の要約や言い換えを提示してきた（「〜という事であっていますか？」）

## 着手前必須チェック

説明を書き始める前に、以下を頭の中で完了させる:

1. **ユーザーの問いを 1 文で再言語化できるか？** できないなら回答前に確認する
2. **結論は 1〜2 文で言えるか？** 言えないなら問いを誤解している可能性が高い
3. **答えが「採用方式名」「典型パターン名」だけになっていないか？** （= 同義反復）
4. **過去の自分の発言と矛盾しないか？**

## 手順

### Step 1. 質問の意図を再言語化

ユーザーの問いを内部で 1 文に圧縮する:

- 「なぜ X が必要なのか」→ 「X が必要となる根本前提を答える」
- 「Y はどうやって動いているのか」→ 「Y の責務と入出力を答える」
- 「Z はあっていますか」→ 「Z の真偽を最初の 1 語で答える」

意図に確信が持てなければ、答える前に **ユーザーに確認** する:

> 「『〜』は『△△ということ』で合っていますか？ 認識が合えば回答します」

### Step 2. 結論を最初の 1〜2 文で先出し

- 結論を **最初の 1〜2 文** で書く
- 詳細・典型例・周辺情報・実装名から書き始めない
- 主語を明示する（「これは」「Presigned URL は」等）
- 「あっていますか」と聞かれたら最初の 1 語は **「あっています」「いえ」「部分的に」** のいずれか

### Step 3. 同義反復チェック

「なぜ X？」に対して以下のような回答は **禁止**:

- 「X 方式を採用しているから」
- 「X という設計だから」
- 「X が必要だから」

これらは問いを言い換えているだけで根拠になっていない。**1 段深い根拠（仕様・制約・前提）まで遡る**:

- ✗「URL 発行が必要なのは Presigned URL 方式だから」
- ◯「URL 発行が必要なのは S3 が SigV4 署名を要求し、ブラウザに credentials を持たせたくないから」

### Step 4. 設計選択肢の提示は「比較表先行」

複数の選択肢を提示するときは、最初に **比較表だけ** を出す。各案の詳細・典型例・実装手順は、ユーザーが選んでから:

```
| 観点 | A 案 | B 案 |
|---|---|---|
| ◯◯ | ... | ... |
```

詳細を全部並べてから「どれにしますか？」とやらない。情報量で判断を麻痺させない。

### Step 5. 詳細展開はオンデマンド

- 1 段目の説明で **「クリアか、深掘りが必要か」** を確認する
- ユーザーが追加要求しない限り、深掘りしない
- 「念のため補足」「ちなみに」「典型例として」で勝手に拡張しない

### Step 6. 矛盾自己 check

説明の中で複数の理由・根拠を並べたら、それらが互いに整合するか **書いた直後に self-review** する:

- 「A 案を採用していた理由」と「A 案を却下する理由」が同じ前提に立っていないか
- 「必要」と「不要」を文脈なしで併記していないか

## アンチパターン

| ✗ NG | ◯ OK |
|---|---|
| 周辺情報・典型例から書き始める | 結論を最初の 1〜2 文で出す |
| 採用方式名で「なぜ」に答える | 仕様・制約・前提まで遡る |
| 設計議論で全案の詳細を一気に展開 | 比較表 → 選択 → 詳細 |
| ユーザーの語彙でなく実装語彙で答える | 一般語に翻訳してから実装語を補足 |
| 1 ターンで全情報を出し切ろうとする | 1 段ずつ確認しながら進む |
| 自分の前発言との矛盾を放置 | 書いた直後に self-review |

## トリガー語サイン（ユーザー発言から検知）

ユーザー発言にこれらが含まれたら **論点を外している前提でリセット**:

- 「おちょくって」「ふざけて」
- 「なぜ確信を」「同じ話」「意味がわからない」
- 「読んでいられない」「長い」「冗長」
- 「論点を外している」「整合性がない」
- 「いえ、ちがう」「そうではない」

検知時のリカバリ手順:

1. **即座に謝罪**（言い訳しない）
2. **ユーザーの問いを 1 文で再言語化** して提示
3. ユーザーが「合っている」と答えたら、結論 1〜2 文で再回答
4. 自分の認識がズレていた経緯を **1 文だけ** 添える（「実装パターンに引きずられていました」等）

## ユーザーの語彙への翻訳

実装語彙（AWS 用語、設計パターン名、フレームワーク用語）をそのまま使うのは禁止。以下のように翻訳する:

| 実装語彙 | 一般語 |
|---|---|
| コントロールプレーン / データプレーン | 「制御の通信」「データの通信」 |
| Presigned URL | 「事前に署名された通行手形 URL」 |
| 2-Phase Commit | 「全部揃ってから一括確定する仕組み」 |
| アンチパターン回避 | 「これだと壊れるので避ける」 |

実装語彙が必要なら、一般語で説明した **後** に「これは技術用語で〇〇と呼ばれる」と補足する。
