# 俯瞰レビュー観点チェックリスト

peer-review は**他者の PR** を見るため、コード品質や行レベルの指摘は除外し、**俯瞰的な 5 観点軸**で評価する。

## 観点軸

### 1. Security（セキュリティ）

- 認証・認可の欠落・不適切な境界
- 秘密情報（Secrets）の不適切な扱い（コミット、ログ、API 露出）
- 外部入力のバリデーション漏れ（インジェクション、SSRF、XSS）
- 暗号化の欠落・不適切なアルゴリズム
- DNS/メール認証（SPF/DKIM/DMARC）の設計不備
- 権限境界の越境（例: 本来 infra-platform の責務を macaron が越境）

### 2. Architecture（アーキテクチャ・設計）

- 責務分離の破綻（レイヤー混在、同じ情報の二重管理）
- モジュール境界の不適切な設計
- 依存関係の方向性（Clean Architecture 違反等）
- 再利用可能なパターンの未活用
- 新旧実装の混在（マイグレーション計画の欠落）
- 拡張性・保守性の欠落
- 代替実装との比較検討の欠如

### 3. Goal Achievement（目的達成）

- Epic / Issue の完了定義に対する網羅度
- ユーザーストーリー・ビジネス要件に対する充足度
- PR 本文の「達成したいこと」と実装内容の整合
- 後続 Issue が着手できる粒度に達しているか（設計 PR の場合）
- 非ゴールの逸脱（スコープクリープ）

### 4. Alternatives（他に最善の案はないか）

- 採用された方式の代替案は検討されたか
- トレードオフが明示されているか
- より簡潔・標準的な方式が存在しないか
- 業界標準から逸脱している場合の理由が妥当か
- 将来の拡張に対する柔軟性

### 5. Spec Consistency（Spec・ADR 整合）

- ADR で決定された方針と整合しているか
- Spec（仕様書）の要件と整合しているか
- CLAUDE.md / Steering ファイルの規約に従っているか
- Epic / Issue の備考（非機能要件、命名規則、制約）と整合しているか
- 既存の同類ドキュメント・実装と矛盾していないか

---

## 除外する観点

以下は peer-review では**意図的に除外**する（他ツールの守備範囲）:

- コードスタイル・フォーマット（lint / formatter の役割）
- 行レベルの細かい typo・文言修正
- テストの実装詳細（self-review の役割）
- 確信度の低い推測的な指摘
- 変更されていない行への提案（scope out）

## ドキュメント PR 特有の観点

ドキュメントのみの PR（設計書、ADR、運用手順書）では追加で以下を評価:

- 他ドキュメントとの相互参照の整合性（リンク切れ、相互引用の一致）
- 用語の一貫性（プロジェクト内での統一）
- バージョン表記・更新日の適切さ
- 図表（Mermaid 等）と本文の内容一致
- サンプルコード（Terraform / YAML 等）と実装の一致
- 責務分離: 他の設計書と守備範囲が重複していないか

## インフラ PR 特有の観点

インフラ（Terraform / CloudFormation / Terragrunt）の PR:

- Provider バージョンの固定（`~> X.Y`）
- State 管理の安全性（import / remove の副作用）
- セキュリティ規約（暗号化、公開範囲、削除保護）
- コスト影響（最小構成、無駄なリソース）
- 環境別（Dev/Stg/Prod）の整合と差分

## アプリ実装 PR 特有の観点

- Clean Architecture の層境界
- 既存の実装パターン（ユーティリティ関数、エラーハンドリング）
- DB スキーマ変更の後方互換性
- API 変更のブレーキング度合い

## 観点マトリクス（軸 × PR タイプ）

レビュー時はこのマトリクスでチェック漏れを防ぐ:

| | ドキュメント PR | インフラ PR | アプリ実装 PR |
|---|:---:|:---:|:---:|
| Security | △ | ◎ | ◎ |
| Architecture | ◎ | ◎ | ◎ |
| Goal Achievement | ◎ | ○ | ◎ |
| Alternatives | ○ | ○ | ◎ |
| Spec Consistency | ◎ | ◎ | ◎ |

◎ = 必須、○ = 推奨、△ = 該当時のみ
