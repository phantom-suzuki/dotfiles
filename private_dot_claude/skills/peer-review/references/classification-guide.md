# 指摘分類ガイド

peer-review で収集した指摘を **5 分類** に振り分けるためのルール。

## 分類定義

| 分類 | 意味 | PR への影響 |
|---|---|---|
| **must-fix** | 本 PR 内で対応が必要（設計の核心 or 実装との不整合） | 対応されないと approve しない |
| **should-fix** | 推奨される対応だが、後続 Issue での追跡も可 | 対応要望は伝えるが blocking しない |
| **question** | 確認したい点、設計意図の明確化要望 | 回答次第で分類が変わる |
| **nit** | 軽微な提案（無視しても問題なし） | 参考として言及 |
| **praise** | 良かった点（レビュー冒頭に配置） | 心理的安全性の担保 |

## 分類の判断フロー

```
指摘を受け取る
│
├─ 設計の核心に関わる？
│  ├─ Yes → 本 PR 内での対応が必要？
│  │        ├─ Yes → must-fix
│  │        └─ No  → should-fix（後続 Issue OK）
│  └─ No  → 次へ
│
├─ 実装・実データとの不整合？
│  ├─ Yes → must-fix（例: ドキュメントのサンプルと実装が違う）
│  └─ No  → 次へ
│
├─ 確認が必要な点？
│  ├─ Yes → question
│  └─ No  → 次へ
│
├─ 良い点？
│  ├─ Yes → praise
│  └─ No  → nit
```

## must-fix の典型パターン

- **Epic / Issue の備考（非機能要件）との不整合**: 例「Epic 備考のサブドメインパターンと PR 設計が食い違う」
- **ドキュメントのサンプルと実装の不一致**: 例「設計書の Terraform サンプルが `default = ""` だが実装は `default = null`」
- **ADR で決定された方針の違反**: 例「ADR-001 が命名規則を定めているのに別の命名になっている」
- **セキュリティ上の明確な不備**: 例「Secrets が平文でコミットされている」

## should-fix の典型パターン

- **運用上の懸念（後続 Issue で吸収可）**: 例「マイグレーション計画が未記載」「運用詳細が未定」
- **責務分離の破綻（他ドキュメントとの整合）**: 例「ses.md の責務範囲が dns-domain.md と重複」
- **技術的に動くが、ベストプラクティスから逸脱**: 例「DMARC rua の到達性問題」
- **参照先の明確化要望**: 例「ADR-002 の参照先 URL/パスが未記載」

## question の典型パターン

- **設計意図の確認**: 例「なぜ Prod だけルートゾーン直配置？」
- **運用体制の確認**: 例「DMARC レポートを誰が監視する？」
- **スコープ確認**: 例「この項目は本 PR のスコープ？ 後続 Issue？」

## nit の典型パターン

- **未記載項目への軽い言及**: 例「後続 Issue で詳細設計されるならこのままで OK」
- **将来の改善提案**: 例「より良い命名があるかもしれない」

## praise の典型パターン

- **設計の網羅性**: 例「Epic の完了定義を網羅している」
- **考慮された制約の明文化**: 例「Review 環境の Dev 共有の前提が明確」
- **セキュリティ配慮**: 例「メール認証 3 点セットを体系的に整理」

## 分類の格上げ・格下げルール

L1（Claude）と L2（Gemini）で分類が異なる場合:

| L1 | L2 | 統合後 | 判断理由 |
|---|---|---|---|
| must-fix | should-fix | **must-fix** | 重い方を採用（見落としリスク回避） |
| should-fix | must-fix | **must-fix** | 同上 |
| must-fix | question | **must-fix + 確認文言** | 確認形式で must-fix を伝える |
| praise | nit | **praise** | 前向きな解釈を優先 |
| question | should-fix | **question** | 確認が先、対応は確認後 |

## 判定（approve / comment / request-changes）

指摘の組み合わせから PR の最終判定を決定する:

| 条件 | 判定 |
|---|---|
| must-fix ≥ 1 かつ 設計の根本を揺るがす | **request-changes** |
| must-fix ≥ 1 だが文言修正レベル / 確認事項 | **comment**（対応希望を伝え、作成者判断に委ねる） |
| should-fix のみ（must-fix なし） | **comment** |
| 指摘なし or praise のみ | **approve** |
| 判断に迷う | **comment**（コラボレーティブな姿勢） |

### 組織文化による調整

- 日本語ネイティブのチームでは `request-changes` がやや強く感じられる傾向
- 明確な blocker でなければ `comment` を選ぶ方が心理的安全性を保ちやすい
- ただし、明確な security 問題や本番影響がある場合は躊躇せず `request-changes`

## severity との関係

severity（重大度）と category（分類）は独立した軸。

| | must-fix | should-fix | nit | question |
|---|:---:|:---:|:---:|:---:|
| **critical** | SQLi → プレースホルダ化 | N/A | N/A | - |
| **warning** | 実装との不一致 | 運用計画欠落 | - | 設計意図確認 |
| **info** | - | ベストプラクティス逸脱 | 将来の改善 | - |
