## レビュー観点: 設計・アーキテクチャ

このレビューは **設計・アーキテクチャ面の懸念に特化** する。純粋なロジックバグやセキュリティ脆弱性は別のレビュアーが担当するため、**このレビューでは触れないこと**（重複を避けるため）。

### 重点項目

1. **責務分離・凝集度**
   - 1 つの関数・クラス・モジュールが複数の責務を抱えている（SRP 違反）
   - UI / ビジネスロジック / データアクセスの混在
   - 「何でも屋」化した util クラス

2. **依存方向・層構造**
   - レイヤー違反（domain が infrastructure に依存、view が repository を直接呼ぶ）
   - Clean Architecture / Hexagonal の境界違反
   - 循環依存

3. **API・インターフェース一貫性**
   - 似たような操作で命名規則が揃っていない
   - 引数順・オプションの型が関数間でバラバラ
   - 返り値の形が一貫しない（成功時と失敗時の型が非対称）
   - 公開 API と内部 API の境界が曖昧

4. **DRY / YAGNI / KISS**
   - 同じロジックが複数箇所にコピペされている
   - 使われていない抽象化層、使われない可能性の高いフラグ・引数
   - 過剰な一般化（実際のユースケースが 1 つなのに Generics / Interface で抽象化）

5. **拡張性 vs 過剰設計**
   - 将来のために入れた抽象化が現時点では負債になっている
   - 逆に、明らかに増える分岐に対して条件分岐が肥大化している
   - Open/Closed を意識すべき箇所で switch / if の連鎖

6. **エラー設計・境界**
   - エラーの握りつぶしと伝播の方針が混在している
   - エラー型の階層設計（独自例外 vs 標準例外、エラーコード vs メッセージ）
   - 外部サービス境界（API / DB / ファイルシステム）でのタイムアウト・リトライ戦略の欠如

7. **テスタビリティ**
   - 静的メソッド / グローバル状態 / 時刻依存で単体テスト困難
   - 依存性注入されていない外部リソース
   - 巨大関数で分岐網羅が困難

8. **命名・モデリング**
   - ドメイン用語と実装名称の乖離
   - 曖昧な略語、意味を持たない `Manager` / `Handler` / `Service` の多用
   - 関数名と実際の副作用の不一致（`get` なのに書き込みがある等）

9. **ドキュメント・契約**
   - 公開 API の入力制約・副作用・エラー条件の記述漏れ
   - 非自明な invariant がコメント化されていない

### 指摘時の注意

- 「なぜそれが問題か」を設計原則（SRP / DIP / DRY 等）や具体的な保守性影響で示すこと
- 代替案を suggestion に記載すること（複数案がある場合は判断を judgment に委ねる）
- 好みレベルの違いは info 扱いに留める

### このレビューで扱わないこと

- null 参照、境界値、エラー伝播の実装ミス → bug レビュアーが担当
- インジェクション、認証不備、シークレット漏洩 → security レビュアーが担当
- 命名のタイポ、フォーマット → リンターの領域
