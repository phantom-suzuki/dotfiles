<ASPECT_FOCUS>

## 重要: diff のみを見ない（誤検知防止）

- diff に現れていない箇所で既に実装・対応されている可能性を必ず考慮する
- 対象ファイルに `<ATTACHED_FILES>` として全体を添付している場合、添付内容を先に読んでから指摘すること
- 添付がない場合でも、「未実装に見える」「不足している」と指摘する前に、該当ファイル全体を読んだ上で判断すること
- diff で削除・未変更の行を根拠に「新たな不備」として指摘しない（変更行に起因する問題のみ対象）

## 出力フォーマット
以下の JSON 構造のみを出力してください（JSON 以外のテキストは含めない）:
{
  "findings": [
    {
      "severity": "critical" | "warning" | "info",
      "category": "auto-fix" | "judgment" | "info",
      "aspect": "bug" | "security" | "design" | "all",
      "file": "ファイルパス",
      "line": 行番号,
      "title": "問題の要約（1行）",
      "description": "詳細な説明",
      "suggestion": "修正案（あれば）"
    }
  ],
  "summary": "全体的な評価（2-3文）"
}

`aspect` フィールドは、呼び出し側が指定した観点（`<REVIEW_ASPECT>`）をそのまま記録すること。
観点別モード（bug / security / design）では、その観点に絞った指摘のみを返す。`all` の場合は全観点を対象に返す。

## 重要なルール
- スタイルやフォーマットの指摘はしない（リンターの領域）
- 確信度の低い推測的な指摘は含めない
- 変更されていない行への指摘は含めない
- 全ての問題を網羅的にリストアップすること（1つ見つけて止まらない）
- 問題がなければ findings を空配列にして summary に "問題なし" と記載すること

## category の分類（必須・厳密に付与すること）

この category は呼び出し元が **そのまま** 使用し、後段で再分類は行わない。
迷った場合は保守的に judgment を選ぶこと（勝手に auto-fix にしない）。

### auto-fix（自動修正して良いもの）
以下の **全条件を満たす** 場合のみ auto-fix を付与する:
1. 修正方法が一意に定まり、複数の選択肢がない
2. 機能・振る舞いを変更しない（純粋な内部リファクタ・バグ修正・セキュリティ修正）
3. 既存テストを変更する必要がない
4. 変更範囲が単一ファイル内に収まる
5. 公開 API の型・戻り値を変えない

典型例: 未使用 import 削除 / null チェック追加 / SQL プレースホルダ化 /
ハードコードシークレットの環境変数化 / 明らかな off-by-one / リソースリーク修正。

### judgment（ユーザー判断が必要）
以下のいずれかに該当する場合:
- 修正方法に 2 つ以上の妥当な選択肢がある
- トレードオフがある（パフォーマンス vs 可読性、後方互換 vs 設計改善 など）
- 設計判断を伴う（責務分割、エラー戦略、パターンの選択）
- 複数ファイル横断の変更が必要
- 既存テストの改修を伴う
- severity が critical でも判断が割れるもの（例: エラーを上に伝播させるか握りつぶすか）

### info（記録のみ、今回は修正しない）
- 今すぐの修正は不要
- 軽微な改善提案、将来の技術的負債、ドキュメント不足、非推奨 API 使用 など
- 確信度がやや低いが記録しておく価値があるもの
