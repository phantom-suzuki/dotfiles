以下の diff をコードレビューしてください。

## レビュー観点
1. バグ・ロジックエラー（確実に誤った結果を生む問題）
2. セキュリティ脆弱性（インジェクション、認証不備、ハードコードされたシークレット等）
3. エラーハンドリングの漏れ
4. エッジケースの未処理
5. パフォーマンス問題（N+1クエリ、不要なループ等）
6. 設計上の懸念（責務の混在、不適切な依存関係等）

## 出力フォーマット
以下の JSON 構造のみを出力してください（JSON 以外のテキストは含めない）:
{
  "findings": [
    {
      "severity": "critical" | "warning" | "info",
      "category": "auto-fix" | "judgment" | "info",
      "file": "ファイルパス",
      "line": 行番号,
      "title": "問題の要約（1行）",
      "description": "詳細な説明",
      "suggestion": "修正案（あれば）"
    }
  ],
  "summary": "全体的な評価（2-3文）"
}

## 重要なルール
- スタイルやフォーマットの指摘はしない（リンターの領域）
- 確信度の低い推測的な指摘は含めない
- 変更されていない行への指摘は含めない
- 全ての問題を網羅的にリストアップすること（1つ見つけて止まらない）
- category の判断基準:
  - auto-fix: 修正方法が1つしかなく、機能変更を伴わない
  - judgment: 複数の選択肢がある、またはトレードオフがある
  - info: 今すぐの修正は不要だが、将来的に検討すべき事項
- 問題がなければ findings を空配列にして summary に "問題なし" と記載すること
