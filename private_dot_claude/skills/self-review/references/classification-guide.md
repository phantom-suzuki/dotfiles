## 分類の判断フロー

```
指摘を受け取る
│
├─ 修正方法が1つだけ？
│  ├─ Yes → 機能変更を伴わない？
│  │        ├─ Yes → auto-fix
│  │        └─ No  → judgment
│  └─ No  → judgment
│
└─ 今すぐ修正が必要？
   ├─ Yes → judgment（上で判定済みでなければ）
   └─ No  → info
```

## auto-fix: 自動修正

**条件**: 修正方法が明確で1つしかなく、機能変更を伴わない。

### 典型的な auto-fix パターン

| パターン | 例 |
|---------|---|
| 未使用 import の削除 | `import { unused } from 'module'` |
| エラーハンドリングの追加 | try-catch の欠落、null チェック追加 |
| 型アノテーションの修正 | 明らかな型の不一致 |
| リソースリークの修正 | close() / defer の追加 |
| 明確な off-by-one エラー | `<= length` → `< length` |
| ハードコードされたシークレット | 環境変数への置換 |
| SQL インジェクション | プレースホルダーへの置換 |

### auto-fix しないもの

- 既存テストの変更が必要になる修正
- API の戻り値の型が変わる修正
- 複数ファイルにまたがる修正（影響範囲が読みにくい）
- パフォーマンス改善（トレードオフがある場合が多い）

## judgment: ユーザー確認

**条件**: 複数の選択肢がある、トレードオフが存在する、設計判断を伴う。

### 典型的な judgment パターン

| パターン | ユーザーに提示する選択肢の例 |
|---------|--------------------------|
| 設計パターンの選択 | 「Strategy パターンに変更する / 現状のまま if-else で対応する」 |
| エラー戦略 | 「エラーを上位に伝播させる / ここでリカバリする」 |
| パフォーマンス vs 可読性 | 「キャッシュを導入する（複雑度UP）/ 現状を維持する（遅いが明快）」 |
| 責務分割 | 「この関数を分割する / 現状の凝集度を維持する」 |
| 後方互換性 | 「非推奨にして新APIを追加 / 既存APIを破壊的に変更」 |
| テスト追加 | 「このエッジケースのテストを追加すべきか」 |

### judgment 項目の提示ルール

1. **背景**: なぜこの指摘が出たか（レビュアー名 + 指摘理由）
2. **選択肢**: 具体的な選択肢を 2-3 個提示
3. **推奨**: もし明確な推奨があれば、理由とともに提示
4. **影響範囲**: 各選択肢の影響範囲を説明

## info: 情報記録のみ

**条件**: 今すぐの修正は不要だが、認識しておくべき事項。

### 典型的な info パターン

| パターン | 例 |
|---------|---|
| 将来の技術的負債 | 「このモジュールは肥大化傾向にある」 |
| 非推奨 API の使用 | 「この API は将来バージョンで削除予定」 |
| テストカバレッジ | 「このパスのテストが不足している」 |
| ドキュメント | 「この関数の意図が不明確」 |
| 軽微な改善提案 | 「より良い命名があるかもしれない」 |

## severity との関係

severity と category は独立した軸。

| | auto-fix | judgment | info |
|---|:---:|:---:|:---:|
| **critical** | SQLi → プレースホルダ化 | エラー戦略の選択 | - |
| **warning** | null チェック追加 | パフォーマンス改善 | 非推奨 API 使用 |
| **info** | - | - | 将来の改善提案 |

- critical + auto-fix: 即座に自動修正（ブロッキング問題を放置しない）
- critical + judgment: 最優先でユーザーに確認（「この修正は必須ですが、方法を選んでください」）
- info カテゴリは severity に関わらず記録のみ

## ユーザーへの提示フォーマット

### judgment 項目の表示テンプレート

```
--- judgment #N ---
[severity] file:line - タイトル
レビュアー: {reviewer_name}

説明:
  {description}

選択肢:
  A) {option_a} ← 推奨
  B) {option_b}

影響: {impact_description}
---
```

### 複数の judgment 項目がある場合

severity: critical を先に表示し、その後 warning を表示する。
AskUserQuestion ツールの選択 UI を使い、**1件ずつ** ユーザーに確認する。
