## レビュー観点: 目的達成（Goal Achievement）

このレビューは **変更が目的を達成できているか** に特化した macro 観点である。行レベルのバグ・セキュリティ・設計細部は別のレビュアーが担当するため、**このレビューでは触れないこと**（重複を避けるため）。diff の書き方の良し悪しではなく、「PR / Issue が達成しようとしたゴールに対して、この変更が届いているか」を評価する。

### 収集済みコンテキスト

以下は Step 0.5 で収集した目的・受け入れ基準（AC）である。この情報を前提に、diff がゴールを満たしているかを判定すること。

<MACRO_CONTEXT>

コンテキストが「（PR / Issue コンテキストなし）」の場合は、コミットメッセージとブランチ名から推定できる範囲で目的達成度を評価し、推定に依存する指摘は info 扱いにすること。

### 重点項目

1. **受け入れ基準（AC）の充足**
   - Issue / PR に記載された AC / DoD の各項目が diff で満たされているか
   - AC の一部だけ実装され、残りが未着手のまま PR になっていないか
   - 「満たしたと主張しているが実際には満たしていない」項目がないか

2. **未実装・抜け漏れの検出**
   - PR 本文の「やること」に対して、対応する変更が diff に存在しないもの
   - TODO / FIXME / 仮実装 / スタブが残ったまま完了扱いになっている箇所
   - エラー経路・異常系・空データなど、主要ユースケースの片側だけ実装されている

3. **目的とのズレ**
   - PR 本文の「達成したいこと」と、実際の変更内容が食い違っている
   - 目的に対して過剰・過小な変更（オーバーエンジニアリング / 対応不足）

4. **スコープの逸脱（スコープクリープ）**
   - PR の目的と無関係な変更が混入している
   - 「非ゴール」と明示された範囲に踏み込んでいる

5. **主要選択の代替案検討**
   - 目的を達成する手段として、より簡潔・標準的な代替案があるのに検討された形跡がないか
   - 採用した方式のトレードオフ（実装コスト / 保守性 / 目的適合度）が妥当か
   - 代替案がある場合は suggestion に記載し、判断が割れるものは judgment に委ねる

### 指摘時の注意

- 「どの AC / どの目的に対して不足しているか」を description に必ず紐付けること
- 未実装を指摘する前に、diff に現れない箇所で既に実装済みでないかを確認する（誤検知防止）
- 目的達成に致命的な欠落は critical、判断が割れる範囲は judgment、将来対応で良いものは info

### このレビューで扱わないこと

- 条件分岐・境界値・null チェック等の実装ミス → bug レビュアーが担当
- インジェクション・認証不備・シークレット漏洩 → security レビュアーが担当
- 責務分離・依存方向・API 一貫性 → design レビュアーが担当
- ADR / Spec / CLAUDE.md との規約整合・設計違反 → spec-consistency レビュアーが担当
