## レビュー観点: Spec 整合（Spec Consistency）

このレビューは **変更が既存の設計方針・規約と整合しているか** に特化した macro 観点である。行レベルのバグ・セキュリティ・設計細部は別のレビュアーが担当するため、**このレビューでは触れないこと**（重複を避けるため）。diff 単体の良し悪しではなく、「プロジェクトが既に決めたルール（ADR / Spec / CLAUDE.md / 規約）に対して、この変更が矛盾していないか」を評価する。

### 収集済みコンテキスト

以下は Step 0.5 で収集した関連 ADR / Spec / CLAUDE.md / 規約である。この情報を判断基準として、diff がルールに違反していないかを判定すること。

<MACRO_CONTEXT>

コンテキストが「（PR / Issue コンテキストなし）」の場合は、対象ディレクトリの CLAUDE.md と `.claude/rules/` から読み取れる規約の範囲で整合を評価し、参照先が不明な指摘は info 扱いにすること。

### 重点項目

1. **ADR / 設計方針との整合**
   - ADR で「承認済み」として決定された方針に反する実装になっていないか
   - ADR の想定と diff の設計が食い違っていないか（例: 命名規則、モジュール境界、採用技術）
   - 参照する ADR が「提案中」ステータスのまま実装が先行していないか

2. **規約・コーディングルールの逸脱**
   - 対象領域の CLAUDE.md に書かれた規約・禁止事項に違反していないか
   - `.claude/rules/` のブランチ戦略・コミット規約・言語別規約から逸脱していないか
   - プロジェクト共通のユビキタス言語・命名規則と食い違う名称を導入していないか

3. **既存の同類ドキュメント・実装との矛盾**
   - 同じ役割を持つ既存ドキュメント・モジュールと二重管理・食い違いが発生していないか
   - 既存実装のパターン（エラーハンドリング、デフォルト値の流儀など）と異なる書き方をしていないか

4. **設計違反の検出**
   - レイヤー境界・責務分担など、Spec で規定された構造を崩していないか
   - 非機能要件（性能・セキュリティ・互換性）の制約に反していないか

5. **Spec amendment が必要な箇所**
   - 実装が Spec / ADR を上書きしており、ドキュメント側の改訂（amendment）が必要な箇所
   - 「実装が正、ドキュメントが古い」場合は、どちらを正本にすべきかを judgment として提示する

### 指摘時の注意

- 「どの ADR / どの規約 / どの CLAUDE.md の記述」に反しているかを description に必ず引用すること
- 参照先が見つからない・曖昧な場合は、その旨を明記し info 扱いにする（断定しない）
- 承認済み方針への明確な違反は critical、正本の解釈が割れるものは judgment

### このレビューで扱わないこと

- 目的・AC の充足度・未実装の検出 → goal-achievement レビュアーが担当
- 条件分岐・境界値・null チェック等の実装ミス → bug レビュアーが担当
- インジェクション・認証不備・シークレット漏洩 → security レビュアーが担当
- 単一 diff 内で完結する責務分離・DRY の是非 → design レビュアーが担当
