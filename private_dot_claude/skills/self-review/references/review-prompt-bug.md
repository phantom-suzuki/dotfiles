## レビュー観点: バグ・ロジック

このレビューは **バグとロジックエラーの検出に特化** する。セキュリティ脆弱性や設計・アーキ面の懸念は別のレビュアーが担当するため、**このレビューでは触れないこと**（重複を避けるため）。

### 重点項目

1. **条件分岐の誤り**
   - 反転した条件、`if` / `else` の取り違え、`&&` / `||` の取り違え
   - 比較演算子の誤用（`==` と `===`、`>` と `>=` 等）
   - 三項演算子の分岐ミス

2. **境界値 / off-by-one**
   - ループ境界（`<` vs `<=`、開始インデックスの誤り）
   - 配列・スライス・リストの端点処理
   - ページング、オフセット、limit の計算

3. **null / undefined / 空値の扱い**
   - null 参照の可能性がある箇所で未チェック
   - オプショナルチェーニング漏れ
   - 空配列・空文字列に対する処理漏れ

4. **エラーハンドリングの漏れ**
   - 例外を握りつぶしている（catch ブロックが空、log だけで return なし）
   - エラー伝播が途中で切れる
   - defer / finally での後始末漏れ
   - 非同期処理の失敗経路

5. **非同期・並行処理の順序**
   - await 忘れ
   - Promise の解決順序に依存した壊れやすいコード
   - 競合状態、共有状態の排他漏れ

6. **リソースリーク・副作用**
   - ファイルハンドル、DB コネクション、タイマー、イベントリスナーの解放漏れ
   - グローバル状態の意図しない変更

7. **型・データ整合性**
   - 想定外の型が来たときの挙動
   - 数値オーバーフロー、浮動小数点の精度
   - 文字列エンコーディング（UTF-8、バイト長 vs 文字長）

### 指摘時の注意

- 「バグである」と断定できる根拠を description に含めること
- 再現手順または問題が発生する入力例を示すこと
- 「可能性がある」「〜かもしれない」レベルの推測は info 扱いに留める

### このレビューで扱わないこと

- SQL インジェクション、XSS、認証不備などのセキュリティ問題 → security レビュアーが担当
- クラス設計、責務分離、API 設計の是非 → design レビュアーが担当
- 命名、コメント、フォーマット → リンターの領域
